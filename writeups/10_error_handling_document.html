Error Handling Without Exceptions

You already know this idiom

You've used this pattern before, without naming it:

if (!(cin >> x)) {
    cout << "input failed" << endl;
}

cin doesn't throw an exception when input fails. It quietly sets an internal flag, and it's on you to check it. C's old errno variable works the same way. We haven't covered exceptions yet, so our classes (Rational, MCounter, and similar) use this exact same idiom: a hidden flag, and functions to check and clear it.

The three functions

is_error() — checks whether the object's data is currently in a bad state (e.g. denominator is 0, count is too big). Recomputes the answer every time it's called; doesn't just remember the past.

error() — returns the flag that was set the last time something went wrong. Doesn't recompute anything, just reports the stored state.

reset() — clears the error flag and puts the object back to a safe, default state (like clearing cin after a bad input).

The one rule that matters most

Nothing forces you to check for an error. Unlike an exception, a forgotten error check will not crash your program or stop execution — it will silently let you keep using a broken object.

a.add_1000();
a.add_1000();   //if 'a' is already in error, you must check first!
b = a + b;      //using 'a' here without checking error() is a bug

Always check error() before you trust an object's value or use it in another operation. This is exactly the discipline exceptions were invented to remove — you'll see that once we get there.
