Rational Number Class

READ ME (Chapter 10, 11)

Rational:

class Rational
{
public:
    Rational();
    Rational(int num, int denom);
    Rational(int w);        //whole number

    //mutators / setters
    void set(int num, int denom);
    void set_num(int n);
    void set_denom(int d);
  
    //accessors / getters
    int get_num() const;
    int get_denom() const;

    //error handling routines
    int error() const;      //return error state
    string error_description() const;   
                            //return a descriptive error message
  
    void reset();           //resets number to 0/1 and clears error state

    //will return object whose error state is set 
    //   if one of the operands is in error. 
    friend Rational operator +(const Rational &num1, const Rational &num2);
    friend Rational operator -(const Rational &num1, const Rational &num2);
    friend Rational operator *(const Rational &num1, const Rational &num2);
    friend Rational operator /(const Rational &num1, const Rational &num2);
    
    //will print UNDEFINED if num is in error.
    friend ostream& operator <<(ostream &outs, const Rational &num);
    friend istream& operator >>(istream &ins, Rational &num);

private:
    int is_error() const;   //checks for error cond and returns error code     
    int _n;                 //numerator
    int _d;                 //denominator
    int _error;             //error code
};

Handling Errors

private function is_error() will check for error conditions (in this case _d==0) It simply returns the error state, it does not set _error. _error can be set like this:

    _error=is_error();

Any function that alters the denominator of the rational number will have to call is_error() to see if _error code must be set. A function that only changes the numerator (such as set_num()) does not need to, since it cannot make the object invalid.

Constructors that cannot possibly produce an invalid object (for example, the default constructor, or the one that takes a single whole number and always sets the denominator to 1) do not strictly need to call is_error(), but they must still explicitly initialize _error to 0 in the member-initializer list. Never leave _error uninitialized.

All operators will have to call the CTOR to return their result. Therefore, if one of the operands is in _error, the operator must ensure the result will be as well.

Important: do not assume the arithmetic itself will naturally produce a zero denominator whenever an operand is invalid. For addition, subtraction, and multiplication this happens to work out, because the two denominators are multiplied together. For division, however, if only the second operand is invalid, the resulting denominator will not necessarily come out to zero on its own. Check is_error() on both operands explicitly, before doing any calculation, in every operator.

Do NOT output error messages from inside any of class functions. Except the insertion operator will output "UNDEFINED" if the rational number is in _error.

The calling function may call rational::error_description() to get a string containing the error message.

Operators are friends of the class, not members of the class.

Do not forget the const and pass by reference on arguments (when needed.)

Do not forget the const keyword on the operator parameters and class functions

 

Testing:

Write an auxiliary function called display_results() and pass it two rational objects. The function will show the result of all the arithmetic operations on the two rationals.

Write a function called test_rational(). This is the function that will be called from main()

In test_rational(), declare two rational objects a and b and assign each a value using a different constructor.

Call display_results() to show the result of all the arithmetic operations.

Then display this menu to the user:

[A] [B]   e[X]it

When the user presses A or B, ask for a rational and input the new values of a or b.

Continue this process until X is inputted.

Make sure to test the case where B (not just A) becomes invalid before a division — this is the one case where an operator that doesn't explicitly check is_error() on both operands can silently produce a wrong, non-UNDEFINED result.
