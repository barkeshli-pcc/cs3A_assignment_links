CS / SFML Project — Architecture Overview

SFML Particle Simulation
Bird's-Eye Architecture

What each class does, how they talk to each other, and why this design makes future projects much easier to build.

The Big Idea

The project separates three very different concerns into three distinct layers. Each layer has one job and one job only. This is what makes the code readable, maintainable, and — most importantly — reusable.

Think of it like a restaurant: the dining room (animate) deals with the customers. The kitchen manager (System) coordinates the cooks. The individual cooks (Particle) each handle their own dish. The kitchen manager never goes to the dining room, and the dining room never touches the stove.

The Three Layers

animate animate.h / animate.cpp SFML Shell

The only class that knows SFML exists. It creates the window, runs the game loop, polls keyboard/mouse events, and controls frame rate. Nothing below this layer needs to know what graphics library is being used.

Every frame does exactly three things: processEvents() → update() → render().

Window creationGame loop Event pollingFrame timing clear / draw / displayEvents → command int
System system.h / system.cpp Simulation Manager

Owns a vector<Particle> and manages the simulation as a whole. Receives a simple integer command from animate and decides what forces to apply. No knowledge of windows, events, or frame timing.

Owns all ParticlesStep(command) Apply forcesDelegates Draw()
Particle particle.h / particle.cpp Physics Entity

Fully self-contained. Knows its own position, velocity, acceleration, mass, and color. Can move itself (Euler integration), bounce off walls, detect collisions, and draw itself when handed a window reference.

pos / velocity / acc / massMove(force) Wall bouncingElasticCollision()Draw(window)
Sidebar sidebar.h / sidebar.cpp UI Utility

A standalone display panel. animate owns it and writes to named slots via sidebar[SB_KEY_PRESSED] = "...". Completely decoupled from the simulation — neither System nor Particle knows it exists.

operator[]draw(window)Display-only

THE KEY INSIGHT:  The only thing that crosses the layer boundary is sf::RenderWindow&, passed as a parameter to Draw(). System and Particle contain zero window-management code — they just accept a reference when it's time to render. This single seam is what makes the whole design swappable.

Reusing This Skeleton for New Projects

Because each layer has a single clear responsibility, you can build a completely different simulation by replacing only the bottom two layers and leaving the SFML plumbing untouched.

✓ Keep As-Is

  • animate.h / animate.cpp
  • sidebar.h / sidebar.cpp
  • constants.h (adjust sizes)
  • The game loop structure
  • Event handling skeleton
  • The command integer pattern

⟳ Replace With Your Own

  • System → Flock, BallPit, Grid …
  • Particle → Bird, Ball, Cell, Agent …
  • Your own physics / rules logic
  • Your own Draw() per entity
  • Your own Step() per simulation
  • New sidebar labels for your data
THE CONTRACT (never changes):

system.Step(int command) — advance the simulation one frame

system.Draw(sf::RenderWindow& window) — draw everything to the window

As long as your new manager class implements these two methods with these exact signatures, animate doesn't care what's inside. You never touch SFML directly in your simulation code.

Starting a New Project — Step by Step

1

Copy the skeleton files

Copy animate.h/.cpp, sidebar.h/.cpp, and constants.h into your new project. These never need to change.

2

Design your entity class

Create Bird, Ball, Cell, etc. Give it Move() and Draw(sf::RenderWindow&). All drawing stays inside this class. This replaces Particle.

3

Design your manager class

Create Flock, Grid, etc. Hold a vector of your entities. Implement Step(int command) and Draw(sf::RenderWindow&). This replaces System.

4

Plug into animate

In animate.h, swap #include "system.h" for your manager's header and change the member variable type. Everything else — the loop, events, sidebar — just works.

5

Wire up keyboard commands (optional)

The command int already flows from key events into Step(command). Add new key cases in processEvents(), handle them in your own Step(). No SFML code needed in your simulation layer.
