Graphing Calculator — Final Project Specification

This project follows the same animate → System → entity layering from the architecture document handed out earlier in this sequence. Here, the entity being managed is a single graph, produced by this pipeline:


┌────────────┐    ┌──────────┐    ┌───────────────┐    ┌─────────────────┐    ┌────────────────┐    ┌─────────────────┐
│  equation  │ -> │  tokens  │ -> │  infix queue  │ -> │  postfix queue  │ -> │  (x, y) pairs  │ -> │  screen pixels  │
└────────────┘    └──────────┘    └───────────────┘    └─────────────────┘    └────────────────┘    └─────────────────┘
Hard constraint: no std:: containers anywhere in this project — no std::vector, std::queue, std::stack, etc. Every container you use must be the Vector, Queue, etc. you wrote in an earlier project. This applies even inside the given code, if you extend it.

What's Given, What You Reuse, What You Build

Given to you

  • animate — owns the window, the run loop, and event → command translation.
  • system — sits between animate and your graph; forwards commands and draw calls.
  • sidebar — a display-only panel with operator[]. Click handling is not built in; see requirement 2 below.

Bring from earlier projects

  • Vector — your own container. Never std::vector.
  • Queue — your own container, used for token streams (infix and postfix).
  • shunting_yard — from your Shunting Yard project. Converts infix to postfix.
  • rpn — from your RPN project. Evaluates a postfix queue for a given x.

Build for this project

  • graph_info — created in animate, passed down through system to graph to plot. You decide its fields.
  • tokenizer — turns equation text into a queue of tokens. A separate document describing token design is coming.
  • coordinate_translator — maps a graph (x, y) to a screen pixel. Entirely your design this time.
  • graph / plot — own the drawing and the point-generation pipeline.
graph_info flows one way: it's created once, in animate, and handed down through system to graph to plot. Nothing downstream makes its own copy — they all read (and update) the same shared object.

Data Flow

One graph is on screen at a time. Entering or selecting an equation replaces it — this project does not overlay multiple graphs.

Memory Management

Every time an equation is tokenized, you allocate a fresh set of token objects. It's easy to accidentally allocate new tokens again on a later pass instead of reusing the ones you already built — and since plot can re-run for every point, on every frame, while zooming or panning, that mistake compounds fast. If nothing ever deletes the old tokens, the program leaks memory the entire time it runs, even though it still looks like it's working correctly.

Two places to check carefully:

  • Re-tokenizing on equation change. Each time set_info() runs a new equation through the tokenizer and shunting yard, decide which object owns the previous queue of tokens, and make sure something deletes them before they're replaced.
  • Re-evaluating the same equation every frame. Zooming and panning re-run the postfix queue through rpn for every point, every frame. This should only read the tokens you already built — it should never call new on a token just to evaluate it.
Test for this directly: leave the program running and pan or zoom continuously for a minute or two, watching memory usage in Task Manager (Windows) or Activity Monitor (Mac). If memory climbs steadily and never levels off, you're leaking tokens somewhere in the pipeline above.

Minimum Requirements

  1. History in the sidebar. When an equation is entered, it's added to the top of the history list.
  2. Click a history entry to graph it. Clicking an equation in the history displays its graph in the work area. The given sidebar only has operator[] — you'll add the click handling and hit-testing yourself.
  3. History persists to disk. On startup, history is read from disk. If no history file exists, one is created, and sin(x) is placed in history as the default entry. File format is your choice.
  4. Natural equation entry. Equations are space-insensitive and case-insensitive — SIN( 1/X ) and sin(1/x) are the same equation. See the equation language section below.
  5. Trig functions. sin, cos, tan, asin, acos, atan must all be supported.
  6. Zoom and pan via domain/range, not SFML. Zooming and panning are achieved purely by changing the graph's domain and range — no SFML zoom/view transforms. See controls below.
  7. Equal-scale square work area. The work area is a square, and x-units and y-units map to the same number of pixels — a circle must look like a circle, not an ellipse.

The Equation Language

Operators

The big five: + - * / ^ — plus the unary minus (e.g. -x, -(x+1)).

Functions

sin, cos, tan, asin, acos, atan, sqrt, abs. Pi is a zero-argument function: Pi().

Multiplication must be explicit

Write 2*x, not 2x. Implicit multiplication is not supported.

sin(2*x)
sqrt(x)+cos(x)
Pi()*x
-1*(x+1)^2
2x              // not valid — write 2*x

Multi-argument functions — extra credit

A multi-argument function takes more than one comma-separated argument, e.g. Max(a, b). Supporting these is worth extra credit; it is not part of the minimum requirements, so don't design around it before the requirements above are solid.

Zoom & Pan Controls

Both should feel natural to use — test them yourself before considering them done. Every one of these works by changing graph_info's domain and range and re-running plot, never by scaling or transforming the SFML view.

  • Enter — zoom in
  • Shift + Enter — zoom out
  • Arrow keys — pan
  • Mouse wheel / trackpad pinch — pan (optional)
