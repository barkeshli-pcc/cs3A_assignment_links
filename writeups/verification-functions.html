Verification Functions

There are times when it is desirable to have a function that will tell us if several parts constitute] a valid whole. This is most easily seen in date, time, fraction, and other types of composite data.

This document details how we design verification functions.

What makes data invalid?

We look for what makes data invalid, not what makes it valid:

Consider date data: m, d, and y.

Date is invalid if m is less than zero or if m is larger than 12.

Date is invalid if d is less than zero or if date is more than the number of days in that month.

Date is invalid if y is less than zero of if y is unreasonably large!

We would NEVER think of putting all these conditions in a single if statement.

It is far easier to have multiple if statements:

 

/**
 * @brief date_is_valid: reutrn true if m/d/y is a valid date
 * @param m
 * @param d
 * @param y
 * @return bool valid code
 */
bool date_is_valid(int m, int d, int y){
    if (m<0) return false;
    if (m>12) return false;
    if (d<0) return false;
    if (d>days_in_month(m, y)) return false;
    if (y<0) return false;
    if (y>3000) return false;
    
    return true;
            
}

Only after checking every condition that would make the data invalid, do we return true.

Notice how easy it will be to modify this function. To add other limitations on the validity of the data.
