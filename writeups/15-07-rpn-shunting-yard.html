Shunting Yard / RPN

Shunting Yard and Reverse Polish Notation (RPN)

1. Overview

This project evaluates ordinary infix math expressions — the way you'd write them by hand, like 3 * (5 + 6 - X) / 9 — including parentheses, a fixed set of built-in named functions, and a single variable. The classic way to do this is in two stages: first reorder the expression into postfix (Reverse Polish Notation) using the shunting-yard algorithm, then evaluate the postfix form with a simple stack machine.

You're given an expression already broken into a queue of typed tokens (numbers, operators, parentheses, function names) — turning a raw string like "3*(5+6-X)/9" into that token queue is not part of this project.

Unlike some earlier projects, malformed input here is not just "rejected" — it must raise a specific, meaningful exception. Silently printing an error and continuing, or crashing, are both incorrect.

2. What It Must Accept / Do

Token types

  • A number — a literal value.
  • An operator — one of +, -, *, /, ^.
  • A left parenthesis / right parenthesis — used for grouping, exactly as in ordinary math notation.
  • A function — a named, built-in operation. Function names are not case-sensitive.

Precedence and associativity

+ and - share the lowest precedence; * and / bind tighter; ^ binds tighter still; a function call binds tighter than any operator. All operators — including ^ — are left-associative. This means 2 ^ 3 ^ 2 evaluates as (2 ^ 3) ^ 2 = 64, not the right-associative convention (2 ^ (3 ^ 2) = 512) some other languages use. This is a deliberate, testable property of this evaluator, not an oversight — don't "fix" it toward right-associativity.

Built-in functions

Each of the following takes exactly one argument, itself possibly a full sub-expression (e.g. sin(x + 1) is valid): sin, cos, tan, cot, log (base 10), ln (natural log), abs.

One function takes zero arguments: pi, which evaluates to the constant 3.14159.

One special name represents the single variable this evaluator supports: whatever name you choose for it evaluates, at call time, to a runtime value supplied to the evaluation call — not to a stored or looked-up value. An expression that never references the variable simply ignores whatever value is supplied.

Multi-argument functions (open design challenge)

Add support for at least one function that takes two arguments, separated by a comma inside one set of parentheses — for example, max(a, b), evaluating to the larger of its two argument values. Each argument may itself be an arbitrary sub-expression, e.g. max(3 + 1, 2 * 5).

This is not exercised by the autograder. It's included because it's a genuinely interesting extension of the pipeline you've already built, and there's real design room in how you choose to handle it — you're encouraged to work out your own approach rather than looking for "the" standard answer.

Error handling

  • A parenthesis mismatch — an unmatched right parenthesis with no corresponding left parenthesis, or a left parenthesis left over once the whole expression has been processed — must raise the provided missing-parenthesis exception. It must not just print a message and keep going.
  • A division whose right-hand operand evaluates to zero must raise the provided divide-by-zero exception. It must not produce infinity, NaN, or a crash.

3. Where It Fits

Infix token queue (already built — not this project's job)
        |
        v
  Shunting Yard   -- reorders tokens into postfix, respecting
        |             precedence, associativity, and parentheses
        v
Postfix token queue
        |
        v
     RPN          -- evaluates postfix with a stack, given one
        |             runtime value for the variable, if used
        v
   double result

Each stage only depends on the type of queue the previous stage produces — not on where the tokens originally came from.

4. Input/Output Contract

Basic precedence, no parentheses:

4 + 2 * 3   -->   10

Parentheses override default precedence:

(4 + 2) * 3   -->   18

Left-associativity, made concrete:

8 / 4 / 2   -->   1      (evaluated as (8 / 4) / 2, not 8 / (4 / 2))
2 ^ 3 ^ 2   -->   64     (evaluated as (2 ^ 3) ^ 2, not 2 ^ (3 ^ 2))

A function call with a compound argument, and the single variable, evaluated with a supplied runtime value:

2 * X + 1     evaluated with X = 5   -->   11
sin(X * 0)    evaluated with X = 7   -->   0

Error cases raise exceptions rather than producing a numeric result:

3 + (4 * 2          -->  raises the missing-parenthesis exception (unmatched "(")
5 / 2)              -->  raises the missing-parenthesis exception (unmatched ")")
4 / (2 - 2)         -->  raises the divide-by-zero exception

A well-formed multi-argument call, each argument itself a sub-expression:

max(3 + 1, 2 * 5)   -->   10

5. Design

Three ideas carry this whole project:

  • Precedence-directed reordering. Walking the infix queue left to right, an operator only gets pushed onto a working stack once every higher-or-equal-precedence operator already sitting on that stack has first been moved to the output. This single rule, applied uniformly, is what turns infix into postfix — including handling parentheses (which simply reset what "already on the stack" means inside them).
  • Evaluating postfix with one stack. A number pushes itself. An operator or function pops however many operands it needs, computes a result, and pushes that result back — the same mechanism works whether it's a plus sign, a trig function, or (if you take on the challenge above) a two-argument function, as long as each one knows its own arity.
  • A comma is a boundary, not an operand or an operator. If you take on multi-argument functions: think about what a comma needs to trigger during the shunting-yard pass so that everything belonging to the current argument is fully resolved before the next argument begins — without disturbing the enclosing function call's own parentheses, which are still needed to know where the whole call ends.

A good sign your evaluator is well-factored: the code that walks the postfix queue and evaluates it shouldn't need a special case for every individual function name — only for how many operands a token consumes and how it computes its result from them.

6. What You Are Building

  • Correct postfix conversion respecting the stated precedence, with every operator — including ^ — treated as left-associative.
  • Correct handling of parentheses for grouping, including nested parentheses.
  • Correct evaluation of every listed built-in function, including when its argument is itself a compound sub-expression.
  • Correct evaluation of expressions that reference the single variable, given a runtime-supplied value.
  • The missing-parenthesis exception raised for any parenthesis mismatch — unmatched close or leftover open.
  • The divide-by-zero exception raised whenever a division's right-hand operand is zero.
  • (Not autograded) Correct evaluation of at least one two-argument function such as max, with comma-separated, independently-evaluated arguments.
